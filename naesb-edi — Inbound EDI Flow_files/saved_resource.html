How a partner's EDI message flows through the gateway

One inbound nomination (X12 873), from the moment a trading partner encrypts it to the moment their receipt is verified. Every stage that can fail still returns HTTP 200 — the failure shows up as a specific error code inside the signed receipt, never as the HTTP status.

01

Prepare the package

  • Compress → sign with their private key → encrypt to our public key
  • Wrap the ciphertext as multipart/encrypted inside input-data
  • Assemble the ordered multipart/form-data envelope (from, to, version, input-format, …)
→POST /inbound · TLS · HTTP Basic Auth→
02

Receive & authenticate

  • Read the body, capture time-c immediately — before anything else
  • Enforce the max body size
  • Check HTTP Basic Auth against this partner's credentials
  • Assign the next sequential trans-id
on failure plain HTTP 401 (auth) — no signed receipt at all
03

Parse & authorize

  • Parse the multipart envelope, unwrap input-data
  • Confirm the envelope's from matches the authenticated partner's DUNS
  • Dedupe by refnum (if this partner uses it) or a digest of the ciphertext
on failure EEDM1xx EEDM701 EEDM121 / GWX‑DUPLICATE‑DIGEST
04

Record it's happening

  • Insert a row in Postgres messages — status = processing
05

Decrypt & verify

  • Decrypt with our private key
  • Verify the signature against this partner's public key
  • Enforce crypto policy — allowed cipher/digest, RSA ≥ 2048 bits
on failure EEDM699 decrypt EEDM604 signature GWX‑WEAK‑ALGO
→fan out concurrently→
06

Deliver the plaintext

  • Filesystem / S3 — durable, keyed by sender DUNS
  • Webhook notification — best-effort, non-durable
on failure 0 durable sinks ⇒ GWX‑SINK‑FAILURE
←≥1 durable sink confirmed←
07

Record & respond

  • Update the messages row — status = accepted
  • Build the receipt: time-c, request-status = ok, server-id, trans-id
  • Detached-sign it with our private key, wrap as multipart/signed
  • Return HTTP 200
←200 OK · signed gisb‑acknowledgement‑receipt←
08

Confirm delivery

  • Verify our detached signature against our public key
  • request-status: ok → done
  • EEDM… / WEDM… → act on it, resend if warranted
no receipt at all retry up to 3× over 30–120 min, then Exchange Failure
trading partner naesb-edi pipeline storage & downstream failure → error code in the signed receipt
What happens after

Audit trail

The Postgres messages row is permanent: status, trans-id, timestamps, and per-sink delivery results, whether the transmission was accepted or rejected.

Downstream processing

The decrypted X12 file sits wherever the sink put it, ready for internal systems to translate and act on. Reading it from there is outside this gateway's scope — its job ends at "delivered somewhere durable."

If nothing comes back

No valid receipt after 3 send attempts spread over 30–120 minutes is a spec-defined Exchange Failure — the partner's system is expected to escalate, not keep retrying silently.
